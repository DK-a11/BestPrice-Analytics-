# Варианты парсинга без нагрузки на серверы источника

## 🎯 Этичные методы парсинга

---

## **1. Использование официальных API**

### Преимущества:
- Легальный способ получения данных
- Стабильная структура данных
- Нет риска блокировки
- Документированные endpoints

### Реализация:
```
- Регистрация в партнерских программах
- Получение API ключей
- Использование официальных SDK
- Соблюдение rate limits
```

---

## **2. RSS фиды и XML выгрузки**

### Описание:
Многие магазины предоставляют XML/RSS фиды для партнеров

### Преимущества:
- Минимальная нагрузка на серверы
- Структурированные данные
- Официально разрешенный метод
- Регулярные обновления

### Источники:
- YML файлы (Яндекс.Маркет формат)
- Google Shopping XML
- Партнерские сети (Admitad, CPA.ru)

---

## **3. Headless Browser с умным throttling**

### Технологии:
- **Puppeteer** (Chrome headless)
- **Playwright** (Multi-browser)
- **Selenium** (Legacy support)

### Механизмы защиты источника:

#### A) Rate Limiting (Ограничение скорости)
```
- Задержка между запросами: 3-10 секунд
- Максимум 10-20 запросов в минуту
- Случайные интервалы (randomization)
- Экспоненциальный backoff при ошибках
```

#### B) User-Agent Rotation
```
- Ротация реалистичных User-Agent
- Имитация различных браузеров
- Randomized headers
```

#### C) IP Rotation и прокси
```
- Использование прокси-сервисов (BrightData, Oxylabs)
- Residential прокси (выглядят как обычные пользователи)
- VPN ротация
- Распределенный парсинг
```

#### D) Сессионное управление
```
- Поддержка cookies
- Имитация поведения пользователя
- Скроллинг страницы
- Случайные клики
```

---

## **4. Кэширование и периодичность обновлений**

### Стратегия:
- **Первичный парсинг**: 1 раз в сутки для всех товаров
- **Приоритетный парсинг**: Каждые 6 часов для популярных товаров
- **On-demand парсинг**: По запросу пользователя (с rate limit)
- **Инкрементальное обновление**: Только изменившиеся данные

### Кэширование:
```
- Redis для временного хранения (TTL: 6-12 часов)
- MongoDB/PostgreSQL для постоянного хранения
- CDN для статических ресурсов
```

---

## **5. Scraping-as-a-Service (SaaS решения)**

### Готовые сервисы:
- **ScraperAPI**: Автоматическая ротация прокси и User-Agent
- **Bright Data (Luminati)**: Enterprise решение
- **Apify**: Облачная платформа для парсинга
- **ParseHub**: Visual scraping tool
- **Octoparse**: No-code scraping

### Преимущества:
- Нет необходимости управлять прокси
- Автоматический обход защиты
- Легальная прослойка
- Масштабируемость

---

## **6. Веб-скрейпинг через RSS агрегаторы**

### Сервисы:
- **FeedBurner**
- **Feedly API**
- **RSS.app**

### Применение:
Создание RSS лент из динамических страниц магазинов

## **7. Гибридный подход (Рекомендуемый)**

### Архитектура:

```
┌─────────────────────────────────────────┐
│         Входной слой                    │
│   (Пользовательский запрос)             │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│    Проверка кэша (Redis)                │
│    TTL: 6-12 часов                      │
└─────────┬───────────────┬───────────────┘
          │ Cache Hit     │ Cache Miss
          │               │
          ▼               ▼
    [Возврат]    ┌────────────────┐
                 │  Router Layer  │
                 └────────┬───────┘
         ┌───────────────┼───────────────┐
         │               │               │
         ▼               ▼               ▼
   ┌─────────┐    ┌──────────┐   ┌──────────┐
   │ API     │    │ XML/RSS  │   │ Headless │
   │ Client  │    │ Parser   │   │ Browser  │
   └────┬────┘    └────┬─────┘   └────┬─────┘
        │              │              │
        └──────────────┼──────────────┘
                       ▼
              ┌─────────────────┐
              │  Data Normalizer│
              └────────┬────────┘
                       ▼
              ┌─────────────────┐
              │   Database      │
              │ (MongoDB/PG)    │
              └─────────────────┘
```

### Приоритизация методов:
1. **Tier 1**: Официальные API (если доступны)
2. **Tier 2**: XML/RSS фиды
3. **Tier 3**: Headless browser с защитой (fallback)

## **8. Соблюдение robots.txt и ToS**

### Обязательные меры:
```
✓ Чтение robots.txt перед парсингом
✓ Соблюдение Crawl-Delay директив
✓ Уважение User-agent правил
✓ Проверка Terms of Service
✓ Использование rel="nofollow" для внешних ссылок
```

### Проверка robots.txt:
```javascript
// Псевдокод
const robotsParser = require('robots-parser');
const robots = robotsParser(url, robotsTxt);
if (robots.isAllowed(targetUrl, userAgent)) {
  // Парсинг разрешен
}
```

## **9. Мониторинг и адаптивность**

### Метрики для отслеживания:
- Успешность запросов (success rate)
- Время ответа сервера
- Количество блокировок (403/429 errors)
- Качество полученных данных

### Автоматическая адаптация:
```
IF (error_rate > 10%):
    increase_delay()
    rotate_proxy()
    change_user_agent()

IF (503_errors):
    exponential_backoff()
    pause_scraping(30_min)
```

## **10. Легальные аспекты**

### Рекомендации:
- ✅ Использовать только публичные данные
- ✅ Не парсить данные за авторизацией
- ✅ Не нагружать серверы интенсивными запросами
- ✅ Указывать источник данных
- ✅ Добавить disclaimer о происхождении данных
- ✅ Получить письменное разрешение (если возможно)

### Дисклеймер на сайте:
```
"Данные о ценах получены из открытых источников
и могут не соответствовать актуальным ценам.
Для точной информации обращайтесь на официальные
сайты магазинов."
```

## 🎯 Итоговые рекомендации

**Для стартапа/MVP:**
1. Начать с официальных API (Wildberries, Ozon)
2. Добавить XML/RSS фиды через партнерские сети
3. Использовать кэширование (Redis) с TTL 12 часов
4. Ограничить парсинг до 5-7 магазинов
5. Настроить rate limiting: 10 запросов/мин

**Для масштабирования:**
1. Внедрить Job Queue (Bull/RabbitMQ)
2. Использовать ScraperAPI для сложных сайтов
3. Построить распределенную архитектуру
4. Добавить мониторинг и аналитику парсинга
5. Масштабировать горизонтально (worker nodes)
